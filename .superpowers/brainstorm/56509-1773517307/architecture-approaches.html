Architecture Approach

How should the frontend and backend relate?

A

Monorepo — Separate Packages (Recommended)

Structure: Single repo, two packages: packages/server (Hono + Socket.io + Prisma) and packages/web (Vite + React Router). Shared types in packages/shared.

Dev: bun dev starts both. Vite proxies API calls to Hono in dev. In prod, Hono serves the Vite build as static files.

Pros

  • Clean separation of concerns
  • Shared types between frontend/backend via oRPC contracts
  • Single deploy artifact (server serves static files)
  • Easy to reason about

Cons

  • Slightly more setup than a single package
B

Single Package — Flat Structure

Structure: One package. src/server/ and src/client/ directories. Vite builds the client, Bun runs the server.

Dev: Same proxy approach. Simpler config, fewer package.json files.

Pros

  • Simplest possible setup
  • No workspace config needed
  • Imports just work — no package boundaries

Cons

  • Easier to accidentally couple frontend/backend code
  • Build config gets more complex in one package.json
  • Harder to add a second client (e.g., mobile) later
C

Two Repos

Structure: Separate repos for server and web. Deploy independently.

Pros

  • Maximum isolation
  • Independent deploy cycles

Cons

  • Overkill for a party game
  • Shared types require publishing a package or copy-paste
  • More repos to manage
